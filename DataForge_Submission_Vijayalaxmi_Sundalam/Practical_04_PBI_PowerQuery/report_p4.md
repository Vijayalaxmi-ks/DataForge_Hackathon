# Practical 4 - Power Query ETL Engine
**Author:** Vijayalaxmi_Sundalam
**Date:** 2026-07-01

## Problem Statement
Establish a self-executing data ingestion pipeline in Power BI that handles variable transactional strings and handles error flags dynamically.

## Procedure
1. Ingested raw source assets directly into the Power BI framework using native text connectors.
2. Cleaned anomalies, handled type casting, and split structural composite ID elements via delimiters.
3. Constructed conditional status parameters and date vectors inside the Power Query interface.
4. Locked back-end configurations by saving the master pipeline.

## Outputs Created
* `p4_power_query_pipeline.pbix` (Power BI binary tracking source states)

## Key Insights
* **Operational Performance:** Tracking the newly engineered `Profitability` column reveals that a minor cluster of high-volume transactions is driving the majority of standard net revenue, masking smaller recurring operational losses.
* **Logistical Profiling:** Splitting the composite `Order ID` exposed distinct distribution hub codes, proving that specific shipping nodes face a higher concentration of structural processing errors.
* **Chronological Cycles:** Isolating the data by the extracted `Year` parameter shows a clear, upward transaction scaling trend across years, pointing to seasonal scaling constraints rather than isolated distribution spikes.

## Conclusion
Deploying a unified Power Query pipeline ensures data integrity, turning messy raw entries into clean, dashboard-ready facts instantly.
